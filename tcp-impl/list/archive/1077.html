   From: Mark Allman <mallman@lerc.nasa.gov>
   Date: Thu, 23 Apr 1998 11:40:24 -0400

   I would say that the initial burst of data caused by a TCP
   connection with an initial window of 4 segments is somewhat similar
   to the initial burst of data caused by 4 parallel connections using
   1 segment initial windows.

Ok.

   But, that is where the similarities end. One connection with an N
   segment initial window is N times more aggressive in the first RTT.
   N connections with an initial window of 1 segment are N times more
   aggressive throughout the transfer.

I totally agree, thanks for pointing this out.

   The change proposed in the I-D alters a _parameter_, not any of
   TCP's fundamental congestion control algorithms.

Really? It depends upon how you look at it.

The majority of the TCP connections seen on the real internet, are
HTTP requests to some server. For the large majority of these
connections, this new initial CWND setting will be the entire data
transfer from the server to the client.

I don't know, I just don't feel very comfortable with the proposals
implicit statement that a flows lower bound should start at 4MSS. If
you haven't done a slow rampup probe of the network capacity and
congestion when you start up, starting with 1 segment in flight as TCP
does now, how can you know this?

I believe this change to be detrimental to the current internet,
and as I've shown this problem can be dealt with satisfactorily on the
client end via immediate ACK of the initial data segment received to
speed up initial slow start, which many stacks are doing already.
