    >>>>> "bf" == Bill Fenner <fenner@research.att.com> writes:
    > FYI, the IPv6 MIB design team, chartered under the IPNG WG, is revising
    > the IP/UDP/TCP MIBs. I'd like to solicit input from implementors on:

    (What kind of implementor do I have to be in order to make my
    contribution eligible for consideration? I only implement networks,
    and sometimes management tools, but no TCP stacks. Hope you don't
    mind my intrusion :-)

    > - The idea of creating a seperate listening table, instead of using
    > the convention of an all-zeroes remote address

    I like it. Most management applications that I can think of handle
    listening and non-listening sockets, ehm, TCP endpoints, separately,
    so having them in separate tables would (1) simplify those
    applications (2) make them more efficient because they don't have to
    skip inappropriate rows and (3) remove the all-zeroes remote address
    convention. Whether that is sufficient justification for the new
    approach I don't know.

    > - Additional counters that would be useful

    Per-connection input/output octet counters would have been extremely
    useful to me sometimes, when trying to look at data rates within
    distributed applications that use TCP.

    Of course there are many other per-connection things that you could
    count (such as segments, retransmissions, duplicate acks, misordered
    segments...) or track (such as window sizes, MSS, whether the
    connection uses SACK, ECN, timestamps, window scaling etc.). It would
    be nice to have all of these accessible via SNMP, but the octet
    counters are the one thing I have actually missed a couple of times.
