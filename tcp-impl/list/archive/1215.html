> As an aside, the BSD stacks also start the 2MSL timer upon entering
> FIN_WAIT_2 state if the application doesn't want to receive any more
> data (e.g. it calls close() ). This prevents the connection from
> hanging forever if the other side doesn't send a FIN.

Right. But I think it can also start the 2MSL timer similarly when the
local side still can receive data. Calling shutdown() to shut down the
send side of a socket that's already in FIN_WAIT_2 is probably the
simplest code path that can do this.

After the discussion here, I am coming to the viewpoint that this is
really an implementation bug more than anything that should have the
attention of this list - it tries to do this only when the local socket
can't do further receives, just not quite hard enough.

The thing that got me started on this is that I was fiddling with the
keepalive parameters (adding sysctls to make them run-time settable)
and found that more than just keepalives used those values; in
particular, I was looking to see what used the tcp_keep* values, and
found them used to set the 2MSL timers as you refer to. The
surrounding code looked wrong enough to prompt my question.
