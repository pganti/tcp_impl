    Hi all,
    I have a question about retransmition timers.
    The question is does each segment that is transmitted need a separate
    timer.

    The reason I ask this is what if the RTT changes between segments ie a
    burst of congestion which bumps up the RTT. Should the timers beyond the
    oldest segment (the timer that is closest to going off) be adjusted to the
    current RTT , technically speaking, or should they stay with the old
    retransmition timers.

    I think the adjusting timers may help avoid unnecessary retransmitions but
    could delay traversing into the retransmition state, causing a slow
    retransmition response. On the other hand it may create quicker
    retransmitions if a packet is lost once coming off a heavily traffic
    period when the old RTT was very high.

    So the implementation of this would mean that you only have one timer and
    a queued sequence of sent segments with time stamps. Once the first
    segment in the queue is acked the timer is cleared, the RTT calculated,
    then the current rtt is used to calculate the time and set the next
    retransmition timer. The time calculation would be along the lines of
    (current RTT) - (the time since the segment was transmitted).

    I do not know what the implications of this approach would be, can anyone
    postulate the implications?

    Ps: RTT meaning Round trip time ( I'm sure you all know that ).

    cheers,
    An inquisitive developer.
