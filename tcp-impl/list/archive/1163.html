I am designing part of a client-server application where it might be handy
to have a very large number of simultaneous active TCP connections. To
what extent is the limit on this a function of the TCP protocol itself,
versus the architecture of particular socket libraries. E.g., suppose I
wanted to write a TCP implementation that could manage 10000 active
connections. What would be the gating factor on today's typical server
platforms? It doesn't seem like memory would be an issue. Would there be
timer limitations, scheduling appropriate retransmissions at a fine enough
granularity to utilize the network efficiently? Is this an area of active
engineering interest, or are the limits imposed by existing implementations
a non-issue for real world applications that are written properly?
