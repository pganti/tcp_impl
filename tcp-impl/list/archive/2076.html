Being involved in the development of a radio network, I had to study the
trace of an HTTP session that performed very badly in order to assess
whether it was due to the radio part of the network. I am attaching the
trace captured on the server, because I would like to have the opinion of
experts on this list to confirm / infirm my conclusions.

Context :
The client (Internet Explorer) is running on a Windows 95 laptop connected
to a radio mobile phone.
The server (Internet Information Server 4) is running on a Windows NT 4 SP6
desktop on an Ethernet LAN.
A tunneling protocol is used between intermediate nodes physically
connected by an Ethernet network ==> MTU is set to 1450 on the client
discounting for the tunnel overhead, in order to avoid fragmentation and
re-assembly at tunnel end-points ==> the client proposes an MSS of 1410
The radio link introduces an important latency that results in a measured
RTT of 9s for a full-size downlink (server to client) packet that is well
over the initial RTO of 3s of TCP implementation of Microsoft. In order to
study the downloading of a large image without being annoyed by this
discrepancy between the assumed RTT and the real one, a persistent HTTP
connection is opened to request a short page first (frame 1 to 14) to train
TCP. Like that, when the request for the large image is issued (frame 15)
the RTO for the TCP connection is 12s i.e. above the real RTT (9s).
Frame numbers appearing in the text below refer to numbers in the attached
trace.

Question / Problem 1 :
The ACK for frame 19 has been lost by the network. Therefore, the content
of frame 19 is repeated 12s later in frame 23. Then ACK for frame 21
arrives (of course also acknowledging frame 19)
The server reacts by sending a packet in sequence with what has been ACKed,
(i.e. SN = end of frame 21 + 1) instead of in sequence with the last packet
sent (i.e. SN = end of frame 22 + 1), hence repeating this packet. Is
this correct, considering that packet in frame 22 was not (for some reason)
a full-size one ?
Remark : from this point the RTO is around 24 s

Question / Problem 2 :
Frame 32 is a repetition of frame 26 after 7s. Because RTO at that moment
is close to 24s we have either :

a buggy early re-transmission but it should not be the case because this
known Windows problem is supposed to be corrected in SP6 for NT 4 that is
in use for the server

or :

a buggy fast retransmit. There are two reasons for which I am saying buggy :
- fast retransmit has been triggered by a second duplicate ACK instead of a
third as mandated by RFC 2581. A confirmation of this lack of compliancy on
the part of Microsoft appears again with frame 43 which repeats frame 37
after the reception of a second duplicate ACK
- the second duplicate ACK is not a duplicate ACK. The ACK sequence we have
is ACK 48714, ACK 48714 and ACK 48855.
I am right on this point ?

Question / Problem 3 :
After frame 26 has been repeated by frame 32, the ACK for frame 26 arrives
(on time). This triggers the repetition of frames 28 and 30 with frames 34
and 35 instead of continuing in sequence with the last packet sent (frame
30). Said otherwise, I would expect the sending of a packet starting with
SN 53085 instead of 50265 for frame 34

Conclusion :
A single lost ACK triggers a cascade of unnecessary repetitions that are
due to :
- a bad implementation of fast retransmit,
- the left side of the window that does not progress as it should.
The problem is self-sustaining as demonstrated by the end of the trace (not
provided in the file attached which has been cut to a reasonable size) due
the following pattern :
- early fast retransmit triggers undue repetitions
- undue repetitions trigger early fast retransmit
Between undue repetitions, the left side of the window progress a little
bit, thanks to what the transfer of the image reaches its end, but only
after a very looong time.
