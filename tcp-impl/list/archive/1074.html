   In conditions where the bandwidth available for each connection is
   high, this seems to be a good idea (reduces latency), but during
   congestion might be a very bad idea.

I've always thought, that this whole problem is much easier to solve
in the client TCP's than on the server side (it is very common for
people to try to rectify a performance problem on the server TCP's
because it's much easier to deploy).

If you ACK immediately the first data frame you receive at the client,
you eliminate many of the startup hassles. I know Solaris and Linux
are doing this now, and it definately helps startup for bulk data
transfers, without pummeling 4 (or whatever ever number is decided
upon) full sized segments all at once into a flow with unknown
congestion conditions. Isn't this almost on par with the web browser
technique of firing up 4 connections to a server simultaneously? It
completely defeats the important congestion control schemes of TCP.
