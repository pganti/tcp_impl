    On Mon, Oct 07, 2002 at 04:08:08PM +1200, Kevin Cross wrote:
    > Hi all,
    > I have a question about retransmition timers.
    > The question is does each segment that is transmitted need a separate
    > timer.

    In the words of Mark Allman [1],

    RFC 2988

    > So the implementation of this would mean that you only
    > have one timer and a queued sequence of sent segments with
    > time stamps. Once the first segment in the queue is acked
    > the timer is cleared, the RTT calculated, then the current
    > rtt is used to calculate the time and set the next
    > retransmition timer. The time calculation would be along
    > the lines of (current RTT) - (the time since the segment
    > was transmitted).

    Your proposal is only a minor change from 2988, where the
    timer is set to expire after RTO + now. Note that in the
    absence of SACK, the sender has no information about whether
    the packet after the one that timed out was received, so it
    would be premature to retransmit it without leving time for
    the ack-of-retransmit.

    -neil
