I was reading over the TCP code in NetBSD preparatory to making some
other changes, and I noticed something odd enough that I have to ask
"is this really the bug it looks like to me?".

Specifically, it appears to me that (at least in that implementation)
once one end of a connection enters FIN-WAIT-2, the peer must keep
sending data, or the connection will be dropped with no attempt to
elicit a response from the peer. As I understand it, FIN-WAIT-2
represents the local end having indicated "no more sends", but nothing
more; in particular, it should be possible to receive data from the
peer. But as I read the code, if the peer remains quiet for long
enough, the connection will be silently destroyed. This is because the
code starts a timer, and if the connection remains idle for a time
appropriate to keepalive expiration, it will be destroyed - but without
any attempt to send anything to find out whether the peer thinks the
connection is still up.

Now, I can see having a timer, to clean up the connection in case the
peer has crashed or something. But tearing down the connection without
checking to see whether the peer thinks it's still up, that looks like
a problem to me. What does the collective wisdom say? Do other TCP
stacks behave similarly?
