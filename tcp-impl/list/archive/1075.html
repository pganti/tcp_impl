> Isn't this almost on par with the web browser technique of firing
> up 4 connections to a server simultaneously? It completely
> defeats the important congestion control schemes of TCP.

I would say that the initial burst of data caused by a TCP
connection with an initial window of 4 segments is somewhat similar
to the initial burst of data caused by 4 parallel connections using
1 segment initial windows. But, that is where the similarities end.
One connection with an N segment initial window is N times more
aggressive in the first RTT. N connections with an initial window
of 1 segment are N times more aggressive throughout the transfer.
For example, if we use 4 connections and we see a loss the effective
window is reduced to 7/8 of the effective window size before loss
occurred (i.e., one of the connections backed off). If a single TCP
connection observed the same loss, the effective window size will be
1/2 the window size when loss was detected (regardless of the
initial window). The change proposed in the I-D alters a
_parameter_, not any of TCP's fundamental congestion control
algorithms.
