>> If the socket is already in FIN_WAIT_2, another shutdown()-for-sends
>> will, as I read it, forcibly both directions (soisdisconnected) and
>> start the 2MSL timer.

> If the socket is already in FIN_WAIT_2, the application must have
> already called close() or shutdown() on it.

True enough.

> Calling shutdown() again certainly seems to disconnect the socket but
> I don't think any reasonable application will cause shutdown() twice
> on a socket.

I disagree. It's entirely plausible to me that two different software
layers could each want to shut down the socket's send side and each be
unaware of the other's existence.

> Ideally speaking though, the second shutdown should be unsuccessful
> rather than disconnecting the socket.

I'd prefer to see it succeed but (since it's already done) do nothing.

I'm definitely not convinced there are no other paths to getting that
timer armed on a socket that can still receive data. I guess my
question for tcp-impl is actually "am I correct that it is
inappropriate to start such a timer on a connection that can still
receive data?".
