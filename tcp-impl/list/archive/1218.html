>> Right. But I think it can also start the 2MSL timer similarly when
>> the local side still can receive data. Calling shutdown() to shut
>> down the send side of a socket that's already in FIN_WAIT_2 is
>> probably the simplest code path that can do this.

> Here's an extract from tcp_input.c in 4.4BSD-Lite code. I assume
> NetBSD is using the same:

> if (so->so_state & SS_CANTRCVMORE) {
> soisdisconnected(so);
> tp->t_timer[TCPT_2MSL] = tcp_maxidle;
> }

> Please note that the 2MSL timer is only started when the sending side
> *cannot* receive more data.

...by that particular code path. The code you quote is functionally
similar to NetBSD's code for transitioning from FIN-WAIT-1 to
FIN-WAIT-2 upon receiving the ACK for our FIN.

> The code for shutdown() will show you that the SS_CANTRCVMORE flag is
> *not* set for shutdown(). Hence 2MSL timer wouldn't be started in
> case of a shutdown().

That doesn't follow (even reading "shutdown()" as "shutdown() for send
only"). It follows that the code you quote wouldn't start the 2MSL
timer in that case, but I never claimed it would. That particular code
path is not a problem.

Follow the calls:

shutdown() -> sys_shutdown [uipc_syscalls.c]
        -> soshutdown [uipc_socket.c]
        -> tcp_usrreq [tcp_usrreq.c], second arg PRU_SHUTDOWN
        -> tcp_usrclosed [tcp_usrreq.c]
        -> if (tp && tp->t_state >= TCPS_FIN_WAIT_2) {
                soisdisconnected(tp->t_inpcb->ip_socket);
                if (tp->t_state == TCPS_FIN_WAIT_2)
                        TCP_TIMER_ARM(tp, TCPT_2MSL, tcp_maxidle);
        }

Note that all calls are unconditional (except for uninteresting cases
like shutdown() on a non-socket). If the socket is already in
FIN_WAIT_2, another shutdown()-for-sends will, as I read it, forcibly
disconnect both directions (soisdisconnected) and start the 2MSL timer.
(On another point, it seems like an abuse of the nomenclature to use a
timer called 2MSL for something other than 2*TCPTV_MSL timeouts.)
