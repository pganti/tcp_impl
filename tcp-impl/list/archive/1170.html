Ok, let me explictly clarify this, since several people have
pinged me about this in private email:

> > Suppose each has a receive buffer of 8KBytes. Then 10K connections
> > requires 800MBytes from somewhere.
> > Perhaps someone else can suggest where.
>
> Most TCPs (at least, BSD based TCPs) don't pre-allocate receive
> buffers. So, 10K idle connections don't use up any memory for
> buffers, just the memory for the connection state. The usual
> place where buffers sit around awhile is on the send side, when
> TCP is waiting for an ACK of the data.

My *main* point is that most modern TCPs don't pre-allocate the
receive buffer, so if there is no incoming data, there is no
receive buffer. Period.

I'm *not* saying that you don't still need a lot of memory to handle
10K connections:
        o Data sitting on the output queue waiting to be acked
        o Out of order data sitting on the TCP resequencing queue
        o Received but unread data on the receive queue
        o Connection state, file-table entries, etc.

                -David Borman, dab@bsdi.com
