In message <199804232243.PAA38472@bossette.engr.sgi.com>, Sam Manthorpe writes:

>> In particular, consider a case where all N connections are sending at their
>> full window size, W, and compare their congestion response with a single
>> connection sending with window size W*N.
>>
>> If a router drops a datagram to signal congestion, the single connection
>> will do a fast recovery and resume sending with window size of W(N/2).
>> With N connections, one connection will do a fast recover and resume sending
>> with window size W/2 -- so the N connections have an effective window of
>> W((2N-1)/2) or nearly twice as large as the single connection.
>
>Yes, but it's even more complicated than that; the one connection is
>significantly less likely to experience subsequent loss after going
>into fast-recovery than the N connections which make a much more modest
>reduction and continue to overload the network. As a result the N connections
>,
>seen as a whole, are likely to see more lost packets than the single
>connection and are likely to undergo further window shrinkages.

Well, yes and no. Yes because the N connections have reduced sending not
at all, they are presumably good candidates to have packets dropped.

But, unless we deploy something like RED, remember that there are results
suggesting that TCP connections tend to synchronize in the bottleneck
queue, with the result that the smallest TCP connection gets penalized more
often than one would expect. So we may nail one connection until we've
driven it to a tiny window, then start reducing the next connection.

In any case, our behavior will, in no way, correctly approximate that of
the N*W window size single TCP.

Craig
