    Hello!

    > it will advance by one byte. On the 8th, it will retract by 7 bytes.

    Yes.

    Plus, window shrinking happens frequently with window update algorithm
    used at receiver side f.e. by freebsd. Look into the archives of this maillist,
    subj "Send window update algorithm".

    > I believe that this is "well tested" in todays Internet (it happens all the
    > time) but nobody has been looking for symptoms.

    All these spurious kinds of window shrinking do not result in _any_
    problems if tcp follows minimal RFC requirements.

    Real differences in behaviour of different implementations (and some problems)
    start when not only effectively delivered to sender advertised window
    is shrinken occasionally, but when two conditions happen simulteneously:
    - receiver shrinks window used internally for segment accepting
      (i.e. _deliberately_ and explicitly)
    - right edge is kept retracted for interval longer than retransmission timeout

    However, this does not affect sender satisfying minimal requirements
    formulated in RFCs.

    Note also, that effect on congestion avoidance (noticed recently in this maillist)
    is present only in implementations not falling to restart cwnd after rto.
    For those wich normally do not reprobe pipe (like linux-2.2), this effect
    is only useful. If AIX and Co do not implement any cwnd validation, it
    is rather their feature. :-)
