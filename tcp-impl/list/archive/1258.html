>>> So what is [the reason I want keepalives]? Connections through a
>>> stateful firewall that drops state if it sees no traffic for a
>>> relatively short period.
>> "I'd like to jam the internet because my firewall is faulty".
> Pardon my flame, but, *what*?????
> You can argue that keepalives cause all sorts of problems, but a
> short packet every two hours on idle connections is going to jam the
> internet? Especially when most connections aren't set for
> keepalives?

Thanks for coming to my defense, Perry, but in this case neither of
those arguments works: I had to set the keepalive interval way down
(four minutes or so) because the damn firewall drops state if the
connection idles longer than that, and I had to create a kernel config
option to force keepalives even on connections that didn't ask for them
because the alternative was to go change damn near every network
program I used to turn on keepalives.

That said, I still don't feel the least bit guilty about "jam[ming] the
internet" with these keepalives. What's the alternative? Application
level keepalives that are probably at least twice as big in bytes and
usually twice as big in packets? And besides all that, what the hell
difference does it make to anyone how I choose to use the bandwidth I
use? I ship around lots of bits that many people would call irrelevant
already, but all of a sudden one tiny packet each way every few minutes
throws you into a tizzy. A week's worth of keepalives is roughly 2500
packets each way - small packets, maybe 64 bytes each? That's around
150K-160K. One background GIF on a web page can equal that - each time
it's used!

And *that* said, I'll note that I do agree the firewall is broken.
Unfortunately replacing it is not a decision I am empowered to make.
