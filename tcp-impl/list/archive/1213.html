>> Now, I can see having a timer, to clean up the connection in case
>> the peer has crashed or something. But tearing down the connection
>> without checking to see whether the peer thinks it's still up, that
>> looks like a problem to me.

> I think someone once suggested that an arbitrary timer would be OK if
> it were only initiated in the close() path as opposed to the
> shutdown() path. Is the code you see executed in both cases, or just
> close()?

I am not certain; I *think* it can be executed via shutdown, calling
tcp_usrreq with PRU_SHUTDOWN, calling tcp_usrclosed, arming the 2MSL
timer. The last step happens only if the socket is already in
TCPS_FIN_WAIT_2, which I'm not convinced can't happen. I *think* every
other code path that arms that timer happens only when user-land has
indiciated no more receives either.

Part of the problem seems to me to be that 793 does not seem to model
the situation where the TCP stack is sufficiently separate from the
data producer/consumer that it's possible for the latter to simply go
away without more than a bare notification thereof to the former; there
seems to be no modeling of the "no more receives possible" state.
