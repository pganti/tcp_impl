Besides what some others have mentioned, delayed ACK'ing policy has a
consequence for a stacks ability to recover from loss quickly.

I experimented for a while with trying to get the delayed ACK as
accurate as possible, such that a lost packet was detected before the
first packet after the "hole" arrived, which gave the sender feedback
information one packet sooner (and thus as a consequence, if this
trick worked, fast retransmit was invoked one packet sooner).

It was hard to get the error in the delayed ack estimation correct so
there weren't any false early ACKs, and everyone keeps telling me that
ACK'ing policies using fine grained timers are a research problem, so
I put this on the back burner for a little while.

But notice how many research papers on loss recovery do not use
delayed ACKs in most of their simulations, ACK'ing every data frame
provides the maximum feedback to the sender, so he can react more
quickly. But not using delayed ACKs has some other performance
consequences.
